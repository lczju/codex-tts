# 项目知识库

## 1. 项目定位

这是一个面向中文 TTS 实验的轻量仓库，目标不是直接提供完整训练系统，而是先搭建一套低成本、可重复、便于试听对比的实验工作流。

当前核心目标：

- 用统一测试文本对比不同 TTS 方法的效果，当前重点是 `CosyVoice` 和 `VITS`
- 预留后续接入 `VALL-E` 类方法的空间
- 在云端 GPU 机器上做推理或小规模 fine-tuning
- 在本地 Windows 环境中整理数据、管理结果、试听对比并记录评分

这个项目目前更像“实验基建 + 展示工作台”，而不是生产级应用。

## 2. 当前仓库实际包含什么

仓库当前主要由四部分组成：

- 文档：实验计划、数据集方案、协作规则
- 前端页面：`web/` 下的本地静态试听与评分页面
- 数据目录：`datasets/` 下的原始数据模板
- 服务器笔记：`server-notes/` 下的 GPU / MobaXterm 使用说明

当前没有发现：

- Python 训练脚本
- 后端服务
- 自动化评测脚本
- 真正的音频样本文件
- 完整的数据处理流水线

也就是说，这个仓库现在仍处在实验准备阶段。

## 3. 技术栈与运行方式

### 前端

- 纯静态页面
- 文件：`web/index.html`、`web/styles.css`、`web/app.js`
- 不依赖构建工具
- 不依赖 npm / node 工程

### 数据加载方式

前端支持两种模式：

- 直接双击打开 `web/index.html`
  - 这种情况下不会 `fetch` 本地 JSON
  - 会使用 `app.js` 内置的默认数据
- 通过本地 HTTP 服务打开页面
  - 会尝试加载 `web/data/results.json`

### 运行特征

- 页面是单文件静态工作台，适合本地试听和人工评分
- 没有登录、数据库、接口、鉴权等能力
- 所有结果数据都保存在 JSON 中

## 4. 前端页面已有功能

`web/app.js` 当前已经实现：

- 测试用例选择
- 按模型筛选结果
- 展示当前测试文本、语言、类别
- 按用例展示模型结果卡片
- 若 `audioPath` 非空则展示音频播放器
- 若音频缺失则提示把文件放到 `web/audio/<model>/`
- 可填写三类主观评分
  - `naturalness`
  - `similarity`
  - `pronunciation`
- 可填写 `notes`
- 支持导入 JSON
- 支持导出当前评分 JSON

页面定位很明确：本地主观试听与记录，不负责训练或推理。

## 5. 数据格式约定

### 测试用例

测试用例由 `testCases` 数组描述，字段包括：

- `id`
- `text`
- `language`
- `category`

当前默认测试集约 15 条，覆盖：

- 中文基础发音
- 数字与时间
- 中英文混合
- 英文句子
- 地名
- 情绪表达
- 语速控制
- 长句稳定性

### 结果数据

结果由 `results` 数组描述，字段包括：

- `caseId`
- `model`
- `audioPath`
- `speaker`
- `checkpoint`
- `generationTimeSec`
- `naturalness`
- `similarity`
- `pronunciation`
- `notes`

其中：

- `caseId` 对应测试用例 ID
- `audioPath` 是网页可访问的相对路径
- 三个评分字段当前允许为空，表示尚未评测

### 当前占位结果

`web/data/results.json` 里目前主要是占位数据：

- 每个 case 默认有两条结果
  - `CosyVoice`
  - `VITS`
- `audioPath` 基本为空
- 评分为空
- `notes` 用来提示后续把真实结果补进来

### metadata 模板

`datasets/raw/speaker_a/metadata.csv` 当前是模板文件，字段为：

- `utt_id`
- `wav_path`
- `text`
- `speaker_id`
- `language`

这说明项目默认按单说话人数据集来组织第一阶段训练样本。

## 7. 项目工作流理解

当前文档描述出的推荐流程大致如下：

1. 本地固定测试文本与 JSON 结构
2. 本地完成试听页面
3. 租用按小时计费的云端 GPU
4. 先跑 `CosyVoice` pretrained inference
5. 再准备单说话人小规模数据做 `VITS` fine-tuning
6. 把云端生成的 `wav` 下载回本地
7. 放入 `web/audio/<model-name>/`
8. 更新 `web/data/results.json` 中的 `audioPath`、评分和备注
9. 在本地页面中统一试听、打分和比较

这里的设计原则很明确：

- 本地只做轻任务
- GPU 机器只做真正需要算力的任务
- 先跑通闭环，再追求复杂工程化

## 8. 成本与资源策略

从文档看，项目明显强调低成本：

- 优先按小时租 GPU，而不是长期持有服务器
- 每次实验后及时关机
- 删除云端实例前先下载音频、日志和 checkpoint
- 本地网页开发不依赖 GPU
- 第一阶段避免引入数据库、多用户系统、长期部署等复杂内容

## 9. 数据集策略

当前文档给出的方向是：

- 第一阶段不建议立刻下载大规模数据集
- 第一轮 `VITS` 微调目标数据量约 `30 分钟 - 2 小时`
- 默认优先公开中文单说话人 TTS 数据集
- 倾向优先尝试 `Biaobei / DataBaker`
- 第一阶段不优先自己录音

这意味着项目的 baseline 目标是：

- 先得到一个可展示、可比较的单说话人中文结果
- 再决定是否扩展到更复杂的数据与模型

## 10. 协作规则

`PROJECT_RULES.md` 给出的约束很重要：

- 面向人阅读的项目文档优先使用中文
- 代码、配置字段、文件名、命令、API 名称优先保持英文
- 推荐使用“中文说明 + 英文术语”的混合写法
- 文件命名优先英文、数字、连字符
- 每次云端实验建议记录日期、平台、GPU、镜像、命令、输入输出位置、耗时、成本和问题

因此后续新增内容时应遵守：

- 文档中文化
- 字段和文件名英文稳定
- 优先保持跨平台、低摩擦目录结构

## 11. 当前状态判断

按现有内容判断，项目大致处于下面这个阶段：

- 已完成：方向设计、目录规划、前端工作台雏形、数据模板、服务器使用笔记
- 未完成：真实推理结果接入、真实训练数据准备、训练脚本、自动化评测、实验日志体系

更具体地说：

- 前端页面已经可以承载真实实验结果
- 但仓库还没有进入“可直接训练”的状态
- 目前更适合继续补齐实验数据和结果，而不是扩展复杂工程

## 12. 已知问题与注意点

### 1. 文本内容存在编码显示异常风险

当前多个 `.md`、`.json`、`.html`、`.js` 文件里的中文在终端输出中出现了乱码表现。原因可能是：

- 文件编码与终端编码不一致
- 历史写入时存在编码问题

需要注意：

- 仓库文件不一定真的损坏，但在不同环境下显示可能不稳定
- 后续如果继续编辑中文内容，最好统一确认 UTF-8 编码

### 2. `web/data/test-cases.json` 可能未被实际使用

当前前端初始化逻辑主要使用：

- `app.js` 内置默认数据
- 或 `web/data/results.json`

`web/data/test-cases.json` 目前看更像独立备份或备用数据文件，不是核心加载入口。

### 3. `results.json` 同时承载两类职责

它既保存：

- `testCases`
- `results`

这让页面加载简单，但会带来重复维护问题。后续如果数据量变大，可能需要明确拆分或统一单一来源。

## 13. 后续协作建议

如果后续要继续推进这个项目，优先级建议是：

1. 统一并修正文本编码，避免中文内容持续乱码
2. 补充 `web/audio/` 目录和真实音频样例
3. 用真实结果更新 `web/data/results.json`
4. 增加实验日志模板，记录每次云端执行细节
5. 再考虑引入训练脚本或自动化评测

不建议当前阶段优先做的事：

- 上来就做后端服务
- 上来就做复杂部署
- 上来就引入大型多说话人数据集
- 上来就扩展过多模型种类

## 14. 一句话总结

这是一个以中文 TTS 对比实验为目标的轻量仓库：本地负责数据整理、试听页面和人工评测，云端负责 GPU 推理与微调，当前已完成实验骨架，但真实音频结果、训练脚本和自动化评测仍未落地。
