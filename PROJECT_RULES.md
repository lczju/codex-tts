# 项目协作规则

## 1. 文档语言

面向人阅读的项目文档优先使用中文，包括：

- 项目计划
- 实验记录
- 服务器操作笔记
- 数据处理说明
- 周计划和任务清单
- 页面展示文案

必要的英文术语保留原文，例如 `TTS`、`VITS`、`CosyVoice`、`checkpoint`、`metadata`、`baseline`、`inference`、`fine-tuning`、`GPU`、`CUDA`、`PyTorch`。

## 2. 代码和配置语言

代码、配置字段、文件名、命令、API 名称优先保持英文。

示例：

- 使用 `caseId`，不要使用中文字段名
- 使用 `audioPath`，不要使用中文字段名
- 使用 `results.json`，不要使用中文文件名

## 3. 中英混合写法

默认采用“中文说明 + 英文术语”的写法。

推荐写法：

```text
先用 CosyVoice pretrained checkpoint 跑 inference，再把结果回填到 web/data/results.json。
```

## 4. 回答语言

项目协作中的回答、说明和总结默认尽量使用中文。

要求：

- 避免出现整段英文回答
- 术语、字段名、命令、路径、模型名可以保留英文
- 如果必须引用英文原文，优先在中文说明中局部引用，而不是整段直接使用英文

## 5. 编码规则

所有新增或修改的文档、HTML、CSS、JS、JSON、CSV 文件统一使用 `UTF-8` 编码。

原因：

- 当前仓库已经出现中文乱码问题
- 文档和前端页面都依赖稳定中文显示
- 如果继续混用编码，后续维护成本会持续上升

执行要求：

- 编辑中文内容时优先确认文件编码为 `UTF-8`
- 不要把中文内容写入 GBK / ANSI 编码文件
- 如果发现旧文件乱码，优先修复后再继续追加内容

## 6. 文件命名

项目文件名优先使用英文、数字和连字符或下划线，避免中文文件名。

推荐：

```text
TTS_BENCHMARK_PLAN.md
PROJECT_RULES.md
server-notes/gpu-setup.md
web/data/results.json
```

## 7. 实验记录要求

每次云端实验建议至少记录这些信息：

- 日期
- 平台
- GPU 型号
- 镜像名称
- 关键命令
- 模型版本或 commit
- 输入数据位置
- 输出音频位置
- 耗时
- 大致成本
- 遇到的问题

记录说明优先中文，命令和报错保留原文。

## 8. 当前推进优先级

方案选择时，优先级如下：

1. 能否快速跑通闭环
2. 学习成本是否可控
3. 经济成本是否可控
4. 是否方便继续扩展

当前阶段不优先引入：

- 后端服务
- 数据库
- 多用户系统
- 长期部署
- 大规模训练平台
