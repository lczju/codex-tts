# 项目协作规则

## 1. 文档语言

面向人阅读的项目文档优先使用中文。

包括：

- 项目计划。
- 实验记录。
- 服务器操作笔记。
- 数据处理说明。
- 周计划和任务清单。
- 面向用户的网页文案。

必要的英文术语可以保留，例如 TTS、VITS、CosyVoice、checkpoint、metadata、baseline、inference、fine-tuning、GPU、CUDA、PyTorch。

## 2. 代码和配置语言

代码、配置字段、文件名、命令、API 名称优先保持英文。

原因：

- 更符合开源项目和深度学习工具链习惯。
- 方便和官方文档、报错信息、第三方仓库保持一致。
- 降低后续接入模型代码时的摩擦。

示例：

- 使用 `caseId`，不使用 `用例编号` 作为 JSON 字段。
- 使用 `audioPath`，不使用 `音频路径` 作为 JSON 字段。
- 使用 `results.json`，不使用 `结果.json`。

## 3. 中英混合原则

默认采用“中文解释 + 英文术语”的方式。

推荐写法：

```text
使用 CosyVoice pretrained checkpoint 先跑 inference，确认云端 GPU 环境可用。
```

不推荐为了全中文而强行翻译成熟术语，例如：

```text
使用舒适声音预训练检查点先跑推断。
```

## 4. 面向用户的回答风格

回答和说明应尽量使用中文。

当涉及命令、路径、依赖、代码、模型名称、字段名时，保留原始英文。

## 5. 文件命名

项目文件名优先使用英文、数字和连字符。

原因：

- 避免 Windows、Linux、远程服务器、Python 脚本之间的编码和路径问题。
- 方便命令行操作。

推荐：

```text
TTS_BENCHMARK_PLAN.md
PROJECT_RULES.md
server-notes/gpu-setup.md
web/data/results.json
```

## 6. 实验记录

每次云端实验建议记录：

- 日期。
- 平台。
- GPU 型号。
- 镜像名称。
- 关键命令。
- 模型版本或 commit。
- 输入数据位置。
- 输出音频位置。
- 花费时间。
- 大致费用。
- 遇到的问题。

这些记录优先写中文，命令和报错保持原文。

## 7. 成本和学习曲线优先级

在方案选择上，优先考虑：

1. 能否快速跑通闭环。
2. 学习成本是否可控。
3. 经济成本是否低。
4. 后续是否方便扩展。

第一阶段避免过早引入复杂工程，例如长期云部署、数据库、多用户系统、自动任务队列、大规模训练平台。
